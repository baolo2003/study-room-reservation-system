# 用户活跃度排行榜 - 面试问题与答案详解

## 一、功能实现逻辑总结

### 核心设计

| 组件 | 作用 |
|------|------|
| **Redis ZSet** | 存储用户活跃度分数，支持 ZINCRBY 原子递增、ZREVRANK 获取排名 |
| **Redis Hash** | 存储幂等标识 `userId:日期 -> {classroomId:seatId -> score}` |
| **双重排行榜** | 日榜（key: `activity_rank_yyyyMMdd`）、月榜（key: `activity_rank_yyyyMM`） |

### 积分规则（ActivityTypeEnum）

| 行为类型 | 积分值 |
|----------|--------|
| 预约（RESERVE） | +1 |
| 签到（SIGN_IN） | +2 |
| 签退（SIGN_OUT） | +3 |
| 取消预约（CANCEL） | +1 |
| 缺席（ABSENT） | -3 |

### 核心流程

```
用户操作(预约/签到/签退/取消/缺席) → 根据ActivityType获取积分 →
检查Hash中是否存在(classroomId:seatId)记录 →
    不存在(首次)：写入Hash + ZINCRBY增加日/月榜分数
    存在且本次负分：删除Hash记录 + ZINCRBY扣分
```

### 关键亮点

1. **幂等性**：用 Hash 记录用户在每个座位的行为，防止重复加分
2. **原子操作**：使用 `incrementScore` 避免并发问题
3. **过期策略**：日榜31天、月榜365天自动过期
4. **复合查询**：一次查询同时获取排名+分数
5. **双重维度**：支持日榜和月榜两种时间维度的排行榜

## 二、面试问题与答案详解

### 基础概念类

#### 问题1：Redis ZSet 是如何实现的？

**答案：**

Redis ZSet（有序集合）的底层实现主要有两种数据结构：

1. **压缩列表（Ziplist）**
   - 当元素数量较少（默认 <= 128个）且元素总大小较小（默认 <= 64字节）时使用
   - 是一种内存紧凑的线性数据结构，适合小数据量场景

2. **跳表（Skip List）+ 字典（Dict）**
   - 当元素数量较多时自动转换为跳表实现
   - **跳表**：支持 O(logN) 的快速查询、插入、删除操作，底层是有序链表+多级索引
   - **字典（Hash表）**：通过 dict 存储 member -> score 的映射，提供 O(1) 的 member 查找

这种组合设计使得 ZSet 既能保证有序性，又能快速通过 member 查分數：

- ZINCRBY / ZADD：O(logN)
- ZREVRANK / ZRANK：O(logN)
- ZSCORE：O(1) 通过字典直接获取
- ZREVRANGE：O(logN + M)

#### 问题2：ZINCRBY 和 ZADD 的区别？

**答案：**

| 命令 | 作用 | 返回值 | 使用场景 |
|------|------|--------|----------|
| **ZADD** | 直接设置成员的分数 | 新添加的元素数量（0或1） | 初始化、批量设置 |
| **ZINCRBY** | 原子递增/递减成员的分数 | 递增后的新分数 | 排行榜积分更新 |

**关键区别：**

1. **语义不同**：
   - ZADD 是"设置为"，直接覆盖原分数
   - ZINCRBY 是"增加"，在原分数基础上累加

2. **原子性**：
   - ZINCRBY 是天然原子操作，不需要 Lua 脚本就能保证并发安全
   - ZADD 在并发下可能导致数据覆盖问题

3. **返回值**：
   - ZADD 返回操作影响的元素数量
   - ZINCRBY 返回递增后的精确分数，便于后续业务使用

**在本项目中的应用：**
```java
// 使用 ZINCRBY 原子递增，支持并发安全
int newans = stringRedisTemplate.opsForZSet()
    .incrementScore(todayRankKey, userId.toString(), score).intValue();
```

#### 问题3：ZREVRANK 和 ZRANK 的区别？

**答案：**

| 命令 | 排序方向 | 排名计算 | 返回值 |
|------|----------|----------|--------|
| **ZRANK** | 升序（从小到大） | 第1名是分数最低的 | 从0开始的索引 |
| **ZREVRANK** | 降序（从大到小） | 第1名是分数最高的 | 从0开始的索引 |

**关键点：**

- ZRANK 返回升序排名（分数最低的排名第1）
- ZREVRANK 返回降序排名（分数最高的排名第1）
- 两者都从 0 开始计数，所以业务展示需要 +1

**在本项目中的应用：**
```java
// 查询用户在日榜的排名（降序，第1名分数最高）
Long rank1 = stringRedisTemplate.opsForZSet()
    .reverseRank(todayRankKey(), userId.toString());
rankItemVo.setRank(rank1 == null ? -1 : rank1.intValue() + 1);
```

### 方案设计类

#### 问题4：为什么用 Hash 做幂等，而不是用 Set？

**答案：**

选择 Hash 而非 Set 的核心原因：**唯一标识的需求**

1. **业务场景分析**：
   - 幂等的目的是防止用户在同一个座位上重复操作
   - 需要识别的是"用户在某一天预约了某个具体座位"
   - 这个标识是复合的：`userId + 日期 + classroomId + seatId`

2. **Hash 的优势**：
   ```java
   // key: activity_rank_用户ID_日期
   // field: classroomId:seatId（座位唯一标识）
   // value: score（该行为的积分）
   
   // 例如：用户1在2024年4月15日预约了教室1的座位A
   key: "activity_rank_1_20240415"
   field: "1:A"
   value: "1"
   ```
   - Hash 可以用复合 field 标识"用户在某座位的状态"
   - 只需要 O(1) 的时间复杂度检查是否存在

3. **如果用 Set 会有问题**：
   - Redis Set 只能存储 member，无法存储额外信息
   - 如果存 "userId:date:classroomId:seatId"，检查时需要完整匹配，无法快速定位
   - Hash 的 O(1) 查询性能更好

4. **内存考量**：
   - Hash 存储 key-field-value 比 Set 更加紧凑
   - 对于大量用户的场景，Hash 内存占用更合理

#### 问题5：如何保证高并发下的分数一致性？

**答案：**

本方案通过以下机制保证高并发下的数据一致性：

1. **Redis 单线程模型**：
   - Redis 是单线程执行命令，所有操作都是原子的
   - ZINCRBY 操作不会被其他命令打断

2. **原子操作 ZINCRBY**：
   ```java
   // 原子递增，无需额外加锁
   stringRedisTemplate.opsForZSet()
       .incrementScore(todayRankKey, userId.toString(), score);
   ```
   - 相比"先查询再更新"的两步操作，ZINCRBY 一步完成，避免并发问题

3. **本地幂等校验**：
   - 在应用层先查询 Hash 判断是否存在：
   ```java
   // 检查是否已存在该座位的记录
   Object obj = stringRedisTemplate.opsForHash().get(useractivitykey, filed);
   if (obj == null) {
       // 首次操作，执行加分
       stringRedisTemplate.opsForHash().put(useractivitykey, filed, String.valueOf(score));
       stringRedisTemplate.opsForZSet().incrementScore(todayRankKey, userId.toString(), score);
   }
   ```
   - 但这种"先查后写"在极高并发下可能存在竞态条件

4. **进一步优化（可选）**：
   - 使用 Redis Lua 脚本将查询和写入合并为原子操作
   - 或者使用 Redis 分布式锁（Redisson）

#### 问题6：如果用户取消预约，如何处理？

**答案：**

代码中已实现完整的取消/扣分逻辑：

```java
else if (ans > 0) {  // 之前存在加分记录
    if (score < 0) {  // 本次是负分（取消/缺席）
        // 1. 删除Hash中的记录（座位状态清除）
        stringRedisTemplate.opsForHash().delete(useractivitykey, filed);
        
        // 2. 扣减日榜和月榜分数
        stringRedisTemplate.opsForZSet()
            .incrementScore(todayRankKey, userId.toString(), score);
        stringRedisTemplate.opsForZSet()
            .incrementScore(monthRankKey, userId.toString(), score);
    }
}
```

**处理流程：**

1. **检查历史记录**：查询 Hash 中是否存在该座位的正向积分记录
2. **执行扣分**：如果存在且本次是负分（ABSENT = -3），执行扣分
3. **清除状态**：删除 Hash 记录，表示该座位不再被占用

**业务场景对应：**

| 场景 | 积分变化 | 处理方式 |
|------|----------|----------|
| 用户正常签退 | +3 | 无特殊处理 |
| 用户取消预约 | +1 → 移除 | 删除Hash记录，扣1分 |
| 用户缺席 | +2 → -1 | 删除Hash记录，扣3分 |

#### 问题7：日榜和月榜数据如何同步？

**答案：**

本方案采用**同步更新**策略，而非事后同步：

1. **每次操作同时更新两个榜单**：
   ```java
   // 更新当天排行榜
   stringRedisTemplate.opsForZSet()
       .incrementScore(todayRankKey, userId.toString(), score);
   // 更新当月排行榜
   stringRedisTemplate.opsForZSet()
       .incrementScore(monthRankKey, userId.toString(), score);
   ```

2. **分数计算方式**：
   - **日榜**：当天累计分数，每天凌晨可能清零或保留
   - **月榜**：当月内所有分数累加（无需额外同步逻辑）

3. **Key 过期策略**：
   ```java
   // 日榜有效期31天
   stringRedisTemplate.expire(todayRankKey, 31, TimeUnit.DAYS);
   // 月榜有效期365天
   stringRedisTemplate.expire(monthRankKey, 365, TimeUnit.DAYS);
   ```

4. **为什么不单独同步**：
   - 同步更新减少了一次 Redis 调用
   - 月榜分数自动就是累积的，无需额外计算
   - 避免定时任务带来的延迟和复杂度

### 扩展优化类

#### 问题8：如何处理 Redis 宕机数据丢失？

**答案：**

这是一个生产环境必须考虑的问题，可从以下几个层面解决：

1. **Redis 自身的高可用**：
   - 部署 Redis 主从复制（Master-Slave）
   - 使用 Redis Sentinel 或 Cluster 保证高可用
   - 开启 AOF 持久化（appendonly yes）

2. **数据降级策略**：
   - 方案一：**定期持久化到数据库**
     ```java
     // 每日凌晨定时任务，将Redis数据写入MySQL
     @Scheduled(cron = "0 0 2 * * ?")
     public void syncToDatabase() {
         Set<String> members = redisTemplate.opsForZSet().range(todayRankKey, 0, -1);
         for (String userId : members) {
             Double score = redisTemplate.opsForZSet().score(todayRankKey, userId);
             userActivityMapper.updateScore(userId, score);
         }
     }
     ```
   - 方案二：**双写策略**
     - 每次更新 Redis 同时写入数据库
     - 缺点：增加系统复杂度，降低性能
   - 方案三：**读取失败降级**
     - Redis 不可用时从数据库查询
     - 牺牲实时性换取可用性

3. **实际建议**：
   - 对于本项目（日榜/月榜），丢失一天数据影响有限
   - 可采用"Redis主从 + 每日备份"的方案
   - 关键业务考虑数据兜底

#### 问题9：如何实现跨天/跨月排行榜？

**答案：**

本方案已通过 key 过期自动处理：

1. **基于 key 的设计**：
   ```java
   private String todayRankKey() {
       return "activity_rank_" + DateUtil.format(new Date(), "yyyyMMdd");
   }
   
   private String monthRankKey() {
       return "activity_rank_" + DateUtil.format(new Date(), "yyyyMM");
   }
   ```

2. **过期策略**：
   ```java
   // 日榜过期时间
   stringRedisTemplate.expire(todayRankKey, 31, TimeUnit.DAYS);
   // 月榜过期时间
   stringRedisTemplate.expire(monthRankKey, 365, TimeUnit.DAYS);
   ```

3. **跨天处理**：
   - 4月15日的日榜 key 是 `activity_rank_20240415`
   - 4月16日自动使用新的 key `activity_rank_20240416`
   - 旧 key 自然过期，无需额外处理

4. **跨月处理**：
   - 月榜 key 是 `activity_rank_202404`（4月）
   - 5月自动切换为 `activity_rank_202405`
   - 月榜分数是当月累积，不需要跨月迁移

5. **历史数据保存（如需）**：
   - 如果需要查询历史排名，需要定时任务将数据迁移到数据库

#### 问题10：排行榜查询量大如何优化？

**答案：**

从以下几个层面优化：

1. **Redis 本身性能**：
   - ZREVRANGE WithScores 是 O(logN + M) 的操作
   - N 是集合大小，M 是返回数量
   - Redis 单线程处理请求，QPS 可达 10W+

2. **读写分离**：
   - 写操作走 Master
   - 读操作走 Slave（读写分离）
   ```java
   // 配置 Redis 读写分离
   @Bean
   public RedisTemplate<String, Object> redisTemplate(RedisConnectionFactory factory) {
       // 配置只读从节点
   }
   ```

3. **本地缓存**：
   - 热点排行榜数据放入 JVM 本地缓存
   - 使用 Caffeine 或 Guava Cache
   - 设置较短的过期时间（如 30 秒）

4. **分页优化**：
   - 当前实现是获取 TopN 后在应用层分页
   - 如果需要深度分页，考虑使用游标（cursor）
   ```java
   // 使用游标分页
   ScanOptions options = ScanOptions.scanOptions().count(100).build();
   Cursor<byte[]> cursor = redisTemplate.execute(new RedisCallback<Cursor<byte[]>>() {
       @Override
       public Cursor<byte[]> doInRedis(RedisConnection connection) {
           return connection.zScan(key.getBytes(), options);
       }
   });
   ```

5. **CDN 加速**：
   - 如果是 Web 端展示，可将 HTML 缓存到 CDN

#### 问题11：如何防止用户刷分？

**答案：**

当前方案已有基础防护，可进一步增强：

1. **现有的幂等防护**：
   - Hash 已防止同一座位重复加分
   - 同一用户对同一座位只能加一次分

2. **业务层面的限制**：
   - 预约次数限制：每人每天最多预约 N 次
   - 签到时间限制：预约后必须在指定时间内签到
   - 座位占用规则：同一用户同时只能占有一个座位

3. **技术手段增强**：
   - **限流**：使用 Redis 计数器限制用户操作频率
     ```java
     // 每分钟最多操作10次
     String limitKey = "rate_limit:" + userId;
     Long count = redisTemplate.opsForValue().increment(limitKey);
     if (count == 1) {
         redisTemplate.expire(limitKey, 60, TimeUnit.SECONDS);
     }
     if (count > 10) {
         throw new RuntimeException("操作过于频繁");
     }
     ```
   - **验证码**：关键操作增加短信/图形验证码
   - **行为分析**：异常模式检测（如短时间内多次取消）

4. **数据库约束**：
   - 唯一索引防止并发插入
   - 事务保证数据一致性

### 场景深挖类

#### 问题12：如果用户同时预约多个座位，如何计分？

**答案：**

当前实现允许用户同时预约多个座位，**每个座位独立计分**：

1. **Hash 设计支持多座位**：
   ```
   key: activity_rank_1_20240415（用户1的2024年4月15日记录）
   field: "1:A" → value: "1"（教室1座位A的预约积分）
   field: "1:B" → value: "1"（教室1座位B的预约积分）
   field: "2:C" → value: "1"（教室2座位C的预约积分）
   ```

2. **ZSet 分数累加**：
   - 用户在日榜的总分数 = 所有座位的积分之和
   - 例如：用户预约了3个座位 → 日榜分数 = 1 + 1 + 1 = 3

3. **业务合理性**：
   - 每个座位代表一次独立的学习行为
   - 多座位 = 多次学习机会 = 更高活跃度
   - 这符合业务逻辑（用户可能预约多个时段）

4. **如果需要限制**：
   - 可在业务层添加判断：检查 Hash 中是否已有记录
   - 如果已有记录，拒绝重复预约

#### 问题14：这个设计方案还有哪些可以改进的地方？

**答案：**

1. **幂等性优化**：
   - 当前"先查后写"在高并发下存在竞态条件
   - 建议使用 Lua 脚本实现原子操作

2. **排行榜边界处理**：
   - 当用户分数为0时，应该从榜单中移除（ZREM）
   - 当前没有处理这种情况

3. **缓存预热**：
   - 日榜/月榜可以在凌晨0点预加载热点数据
   - 避免早高峰并发访问冷启动

4. **分页查询优化**：
   - 当前获取 TopN 后在应用层遍历查用户信息
   - 建议批量查询用户信息，减少 N 次数据库调用

5. **监控告警**：
   - 添加 Redis 内存使用、命中率等监控
   - 异常操作自动告警

## 三、简历写法建议

### 简历描述示例

> **基于 Redis ZSet 设计用户活跃度排行榜**
> - 采用 Redis ZSet + Hash 实现，支持日榜/月榜双维度排名
> - 利用 ZINCRBY 实现分数原子递增，结合 Hash 实现幂等性校验
> - 支持查询用户排名、分数及 TopN 榜单，查询性能达到 O(logN)
> - 覆盖预约、签到、签退、取消、缺席等业务场景的积分计算

### STAR 法则包装

| 要素 | 内容 |
|------|------|
| **S（背景）** | 学习室预约系统需要展示用户活跃度排行，激励用户持续使用 |
| **T（任务）** | 设计并实现实时排行榜功能，支持日榜/月榜，需保证高并发下的数据一致性 |
| **A（行动）** | 使用 Redis ZSet 存储分数，结合 Hash 实现幂等性；处理了并发扣分、跨天数据迁移、负分场景 |
| **R（结果）** | 查询性能从数据库的 O(N) 优化到 Redis 的 O(logN)；支持每日百万级查询 |
